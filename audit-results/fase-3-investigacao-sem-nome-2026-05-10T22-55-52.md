## Hipóteses sobre origem

100% dos docId batem com slugs em `clientes_base/`. Isso sugere fortemente
que estes docs foram criados por algum pipeline que copiou o docId mas não
o campo `nome_cliente`. Candidatos:
- `fecharPeriodo` em firebase.ts (linhas 752-781) — `batch.set(destRef, d.data())`. Se
  o doc de origem em clientes_base/ não tinha nome_cliente em algum momento, o erro
  se propagou. **Improvável** — clientes_base/ atual tem todos com nome.
- `copiarPeriodo` em firebase.ts (linhas 537-567) — mesma operação `batch.set`.
  Possível que um período anterior tenha tido docs sem nome que foram copiados.
- Algum import Excel antigo gravou docs com docId mas sem campo `nome_cliente`.

## Recomendação técnica

**Opção recomendada:** recuperação cruzada com clientes_base/. Para cada doc:
1. Ler `clientes_base/{docId}.nome_cliente` → popular no snapshot.
2. Ler `clientes_base/{docId}.id_estavel` → herdar.
3. Aplicar tudo em um único `updateDoc` por doc.
